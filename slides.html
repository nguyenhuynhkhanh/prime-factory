Dark Factory

Specs in, production-grade features out. A multi-agent framework for Claude Code where no single agent has the full picture — so no agent can cut corners.

Core innovation

Information barriers

Each agent has strict boundaries on what it can see. The code-agent must implement correctly to pass tests it has never seen.

Spec Agent

Sees: codebase, docs, project profile
Blocked: holdout scenarios, test results

Code Agent

Sees: spec, public scenarios, architect findings
Blocked: holdout scenarios, test results

Test Agent

Sees: spec, holdout scenarios only
Blocked: public scenarios, source code changes

Architect Agent

Sees: spec, codebase, code map
Blocked: all scenarios, all tests

Artifacts

Project structure

Everything Dark Factory produces lives under dark-factory/. Temporary artifacts are deleted after promotion — git history is the permanent archive.

dark-factory/
  project-profile.md ← architecture, conventions, quality bar
  code-map.md ← dependency graph, hotspots, entry points
  code-map.mermaid ← visual diagram for humans
  manifest.json ← active features only (cleaned up after promotion)

  specs/
    features/{name}.md ← feature spec (temporary)
    bugfixes/{name}.md ← debug report (temporary)

  scenarios/
    public/{name}/ ← visible to code-agent
    holdout/{name}/ ← hidden from code-agent, used by test-agent

  results/ ← test output (gitignored)

Lifecycle:  active → passed → promoted → cleaned up (removed from manifest, artifacts in git history)

Bug pipeline

Red-green cycle

Strict discipline that prevents the most common AI failure: fixing the symptom instead of the root cause.

Red phase

Write a failing test that proves the bug is real. Zero source code changes allowed.

Green phase

Implement the minimal fix. Zero test file changes allowed. All existing tests must still pass.

Before any fix: 3 investigators run in parallel (code path tracer, history detective, pattern analyst) producing a synthesized report with systemic analysis and regression risk assessment.

Onboarding

Deep code map

Built by parallel scanner agents — one per source directory. Every downstream agent reads relevant sections.

  • Module dependency graph Which modules import from which, grouped by directory
  • Entry point traces Route → controller → service → repo → database
  • Shared dependency hotspots Modules imported by many others — high blast radius
  • Interface boundaries What each module exports as its contract
  • Cross-cutting concerns Middleware, decorators, base classes affecting multiple modules
  • Circular dependencies Detected and flagged for resolution

Output includes a Mermaid diagram (code-map.mermaid) for human visualization.

Scaling

Group orchestration

Large features decompose into multiple specs with declared dependencies. Dark Factory resolves them into execution waves.

CapabilityHow it works
Wave executionFoundation specs complete and merge first, then dependents start from updated code
Parallel worktreesIndependent specs run simultaneously in isolated git worktrees
Resume semanticsRe-run after failure — completed specs are automatically skipped
Failure isolationOnly transitive dependents pause; independent specs continue unaffected
--groupAll specs in a feature group
--allEvery pending spec across all groups, independent groups in parallel
Reliability

Regression prevention

Bug fixes include built-in protection against recurrence. Tests that don't explain what they guard are tests that get deleted.

  • Systemic analysis — search for the same vulnerable pattern across the entire codebase, not just the reported instance
  • Risk classification — isolated, systemic, or shared-code risk with concrete reintroduction vectors
  • Root cause depth — distinguish immediate cause from deeper enabling pattern. Tests target the deeper pattern.
  • Variant scenarios — proportional to risk: high 3–5 variants, med 1–2, low reproduction only
  • Annotated tests — every promoted test carries root cause pattern, guarded code locations, and bug reference

Getting started

Two ways to install

Claude Code Plugin (recommended)
/plugin marketplace add nguyenhuynhkhanh/dark-factory
/plugin install dark-factory@dark-factory-marketplace
npm (per-project)
npx dark-factory init

Then: open Claude Code → /df-onboard → describe what you need.

Plugin installs once and works across all projects with auto-updates. npm installs into the current project only.

Design decisions

Why it works this way

Holdout scenarios Code-agent can't game tests it hasn't seen
3 parallel leads Single agents have blind spots. Three perspectives catch more.
Git worktree isolation Parallel specs without merge conflicts
Strict red-green Prevents changing tests to match broken behavior
Deep code map Agents know blast radius before making changes
Manifest lifecycle Track active work only. Git history is the archive.
Smart decomposition Large features split into dependency-aware waves
Auto-approved permissions Agents run without interrupting developers
