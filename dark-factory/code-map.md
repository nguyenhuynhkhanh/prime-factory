## Cross-Cutting Concerns

- **Information barriers**: Every agent declares explicit NEVER-read constraints. Holdout scenarios hidden from code-agent. Public scenarios hidden from test-agent. Tests/scenarios hidden from architect-agent. Enforced by 50+ test assertions.
- **Code map freshness protocol**: df-intake, df-debug, and df-orchestrate all share the same pre-pipeline hook pattern: read code-map.md header -> compare git hash -> spawn codemap-agent if stale.
- **Slim file ecosystem**: Three agents/skills produce slim files (onboard-agent writes profile-slim, codemap-agent writes code-map-slim, df-cleanup regenerates both). One agent consumes them (architect-agent for Tier 1/2). All share the invariant: slim file `Git hash:` must match the full file.
- **Plugin mirroring**: all agents, skills, rules, and templates must exist in both `.claude/` and `plugins/dark-factory/`. Contracts test enforces exact match. Now includes `project-memory-template.md`.
- **Worktree isolation**: df-intake, df-debug, df-orchestrate, and implementation-agent (multi-track code-agents) all spawn agents with `isolation: "worktree"` for parallel, conflict-free execution. Wave-based dependency resolution in df-orchestrate.
- **Template-driven output**: spec-agent, debug-agent, and onboard-agent all read structured templates from `dark-factory/templates/`. Now includes `project-memory-template.md` (canonical schema for all 8 memory files).
- **Pre-flight test gate**: implementation-agent runs project test suite before architect review. Bypassable via `--skip-tests` (logged in manifest as `testGateSkipped: true`).
- **Tiered architect review**: spec-agent classifies tier (Tier 1/2/3) at spec-write time. implementation-agent reads tier at spawn time. architect-agent uses tier to select context loading strategy and round budget. Self-escalation permitted upward only.
- **Round summarization**: architect-agent writes compact round summaries (max 400 words) after each review round. Next round begins by reading prior summary. Graceful fallback if summary is missing.
- **Memory single-writer protocol**: `dark-factory/memory/` has exactly two authorized writers: onboard-agent (bootstrap only, at onboard time) and promote-agent (ongoing, at promotion time). All other agents are read-only. Parallel code-agents running in worktrees MUST NOT write to memory files. Permanent IDs are assigned exclusively by promote-agent; placeholder IDs (`INV-TBD-*`, `DEC-TBD-*`) used in specs until promotion.
- **Index-first memory load**: spec-agent and debug-agent both follow the same protocol — read index.md first, use tags for domain lookup, load only matched domain shards. Missing index → load all 6 shards (conservative fallback). Missing shard → log and continue (non-blocking).
- **UI Layer / E2E detection (Phase 2.5)**: onboard-agent uses an explicit allowlist for frontend frameworks and E2E frameworks. Records 4 rows in profile Tech Stack table consumed by test-agent for backend-only exclusion.
- **Flaky E2E routing**: test-agent emits `flakyE2E: true` boolean. implementation-agent uses this as the single authoritative signal.
- **Dev server lifecycle (test-agent)**: 3-tier fail-soft cascade: (1) Playwright config webServer, (2) project-profile `Dev Server` command, (3) `npm run dev` fallback.
- **Git hook installation (Phase 7.5)**: onboard-agent detects existing hook infrastructure (Husky, Lefthook, simple-git-hooks) and installs into the appropriate location. Idempotent.

## Dynamic/Runtime Dependencies

- **Agent spawning via Agent tool**: All inter-agent communication happens through Claude Code's Agent tool at runtime. Not statically traceable from file imports.
- **Tier-conditional architect spawn**: implementation-agent reads `Architect Review Tier` from spec at runtime. Decision to spawn 1 combined vs. 3 parallel domain architects is made per-spec.
- **Slim file fallback**: architect-agent probes for slim file existence at runtime (Tier 1/2 only). Falls back to full file silently if slim is absent.
- **Manifest-driven dispatch**: df-orchestrate reads manifest.json at runtime to determine which specs to implement and in what order.
- **Index-first memory shard selection**: spec-agent and debug-agent select which domain shards to load at runtime based on index.md tags. Missing index → fallback to all shards.

- **Serena MCP availability**: code-agent and debug-agent probe Serena tool availability at runtime and fall back to Read/Grep. debug-agent is read-only (never uses Serena mutation tools).
- **Plugin installation targets**: bin/cli.js copies files to arbitrary target directories at runtime.
- **Flaky-E2E routing decision**: implementation-agent reads `flakyE2E` from test-agent results metadata at runtime.
- **Dev server detection**: test-agent's webServer source is determined at runtime by probing playwright.config.*, then project-profile, then npm run dev.
- **Git hook target**: onboard-agent's hook target file is selected at runtime based on detected hook infrastructure in the target project.
