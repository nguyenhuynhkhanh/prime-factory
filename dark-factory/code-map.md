## Cross-Cutting Concerns

- **Information barriers**: Every agent declares explicit NEVER-read constraints. Holdout scenarios hidden from code-agent. Public scenarios hidden from test-agent. Tests/scenarios hidden from architect-agent. NEW: qa-agent never reads holdout scenarios from prior features; coverage-map.json is the ONLY artifact architect reads at Gate 2 — never scenario file contents. Enforced by test assertions. Shared `holdout-barrier.md` block ensures consistency across compiled agents.
- **3-gate architect review pipeline**: Gate 1 (ARCH_SPEC_REVIEW) = spec correctness + mandatory ADR writing; Gate 2 (ARCH_SCENARIO_REVIEW) = coverage map completeness (ADR → scenario mapping); Gate 3 (ARCH_DRIFT_CHECK) = implementation conformance to ADRs. All 3 gates must pass before promotion. Max rounds: Gate 1 = 5, Gate 2 = 3, Gate 3 = 2.
- **Layer 0 → Layer 2 intent chain**: `intent-foundation.md` (project-wide, max 20 decisions, always loaded) → `intent-adr-{spec_id}.md` (per-spec Layer 2 ADRs, written by architect at Gate 1, read by qa-agent for coverage map and by architect at Gates 2/3). Intent propagates from foundation → spec → ADRs → scenarios → implementation.
- **Coverage map contract**: qa-agent writes `coverage-map.json` linking every Layer 2 ADR to scenario IDs. `uncovered_adrs` field is the single signal for Gate 2 pass/fail. Architect at Gate 2 reads ONLY this file — never reads scenario contents. qa-agent's self-review pass ensures completeness before emitting.
- **Path-passing contract (ao-thin-impl-agent)**: implementation-agent passes ONLY explicit paths (specPath, publicScenariosDir, architectFindingsPath) to code-agent — never inline content. findings.md is the intermediary artifact: written by implementation-agent after architect review, deleted during Step 5 cleanup. Compiled implementation-agent must be <= 3,200 tokens.
- **Structured JSON result contract (ao-thin-wave-orchestration)**: implementation-agent always emits structured JSON at every lifecycle exit. df-orchestrate reads ONLY JSON from wave agents. Wave agents are fresh per wave. Malformed results → sentinels (result-parse-error, wave-agent-crash). token-cap is a distinguishable status.
- **Code map freshness protocol**: df-intake, df-debug, and df-orchestrate all share the same pre-pipeline hook pattern: read code-map.md header -> compare git hash -> spawn codemap-agent if stale.
- **Slim file ecosystem**: Three agents/skills produce slim files (onboard-agent writes profile-slim, codemap-agent writes code-map-slim, df-cleanup regenerates both). One agent consumes them (architect-agent for Tier 1/2). Slim file `Git hash:` must match full file.
- **Plugin mirroring**: all agents, skills, rules, and templates must exist in both `.claude/` and `plugins/dark-factory/`. Contracts test enforces exact match. All 10 agent files carry `model-role` frontmatter. Compiled from `src/agents/*.src.md`; build system writes both targets.
- **Compile-time agent assembly**: `src/agents/*.src.md` reference shared blocks via `<!-- include: shared/X.md -->` directives. `npm run pretest` runs build before test to guarantee sync. Contracts test verifies plugin mirror matches built output.
- **Model-role frontmatter**: All 10 agents declare `model-role: generator|judge`. Judge agents (architect-agent, test-agent, promote-agent, implementation-agent) always use claude-sonnet. Generator agents (spec-agent, code-agent, debug-agent, onboard-agent, codemap-agent, qa-agent) use model selected by `--mode` flag.
- **Worktree isolation**: df-intake, df-debug, df-orchestrate, and implementation-agent (multi-track code-agents) all spawn agents with `isolation: "worktree"`. Wave-based dependency resolution. Best-of-N adds track-a and track-b worktrees.
- **Template-driven output**: spec-agent, debug-agent, and onboard-agent all read structured templates from `dark-factory/templates/`.
- **Pre-flight test gate**: implementation-agent runs project test suite before architect review. Bypassable via `--skip-tests`. `--skip-tests` and `--mode` are orthogonal.
- **Tiered architect review**: spec-agent classifies tier at spec-write time. implementation-agent reads tier at spawn time. architect-agent uses tier for context loading and round budget. Self-escalation upward only.
- **Round summarization**: architect-agent writes compact round summaries (max 400 words) after each review round. Next round begins by reading prior summary. Graceful fallback if missing.
- **Design Intent system**: onboard-agent extracts DI candidates (Phase 3.7d) → spec-agent reads DI shards and auto-populates `## Design Intent` with `DI-TBD-*` placeholders → promote-agent materializes into permanent `DI-NNNN` IDs (DI write-through). architect-agent enforces via Intent & Drift Check.

- **Index-first memory load**: spec-agent, debug-agent, and code-agent follow index-first protocol — read index.md first, load only matched domain shards. Shared `memory-index-load.md` block in `src/agents/shared/` ensures consistency.

## Circular Dependencies

None detected. The dependency graph is a DAG:
- Skills spawn agents (never the reverse)
- Agents read shared data files (project-profile, code-map, manifest, memory/) but do not import each other
- implementation-agent spawns other agents but is never spawned by them (only by df-orchestrate)
- codemap-agent is spawned by onboard-agent and pre-pipeline hooks but never spawns non-scanner agents
- spec-agent can be re-spawned by implementation-agent (architect blockers OR flaky-E2E bugfix) but does not spawn implementation-agent — still acyclic
- qa-agent is spawned by implementation-agent (after Gate 1); it writes coverage-map.json which architect reads at Gate 2; architect never spawns qa-agent — no cycle
- architect-agent reads intent-adr-{spec_id}.md (which it wrote at Gate 1) at Gates 2/3; this is self-referential reading, not a cycle
- promote-agent writes to memory/ (including DI shards) but is never spawned by onboard-agent — both write to memory/ independently, no cycle
- build-agents.js reads src/*.src.md and writes .claude/agents/ — no agent reads its own source; no cycle
- wave agents are ephemeral pass-through coordinators: spawn implementation-agents and return JSON results; no cycle with df-orchestrate

## Dynamic/Runtime Dependencies

- **Agent spawning via Agent tool**: All inter-agent communication happens through Claude Code's Agent tool at runtime. Not statically traceable from file imports.
- **Tier-conditional architect spawn**: implementation-agent reads `Architect Review Tier` from spec at runtime. Decision to spawn 1 combined vs. 3 parallel domain architects is made per-spec.
- **Gate progression dispatch**: implementation-agent advances through Gates 1→2→3→4 based on runtime gate results (APPROVED/GAPS_FOUND/CLEAN/DRIFT_FOUND). State transitions are runtime decisions.
- **Pipeline mode dispatch**: implementation-agent receives `--mode` at runtime; selects claude-sonnet vs. claude-opus per mode+tier combination. Judge agents invariantly use claude-sonnet.
- **Best-of-N activation**: quality mode + Tier 3 → two parallel code-agent worktrees at runtime. Neither condition alone activates it.
- **Slim file fallback**: architect-agent probes for slim file existence at runtime (Tier 1/2 only). Falls back to full file silently if slim is absent.
- **Manifest-driven dispatch**: df-orchestrate reads manifest.json at runtime to determine which specs to implement and in what order.
- **Index-first memory shard selection**: spec-agent, debug-agent, and code-agent select which domain shards to load at runtime based on index.md tags. Missing index → fallback to all shards.
- **DI shard availability**: design-intent shard files may be absent (pre-onboard or not yet extracted). Non-blocking at every stage.
- **Memory shard availability**: individual INV/DEC shard files may be absent. Non-blocking.
- **intent-adr availability**: `intent-adr-{spec_id}.md` is written by architect at Gate 1; if missing when qa-agent reads it: `adr_coverage` is empty, `uncovered_adrs` is empty, noted in `self_review_notes`. Gate 2 receives an empty coverage map and approves.
- **Serena MCP availability**: code-agent and debug-agent probe Serena tool availability at runtime and fall back to Read/Grep.
- **Plugin installation targets**: bin/cli.js copies files to arbitrary target directories at runtime.
- **Flaky-E2E routing decision**: implementation-agent reads `flakyE2E` from test-agent results metadata at runtime.
- **Dev server detection**: test-agent's webServer source is determined at runtime by probing playwright.config.*, then project-profile, then npm run dev.
- **Git hook target**: onboard-agent's hook target file is selected at runtime based on detected hook infrastructure.
- **AFK PR creation**: implementation-agent reads spec `## Context` and `## Acceptance Criteria` at runtime to draft PR body; gh pr create is a runtime shell call (non-blocking if it fails).
- **Wave agent JSON parsing**: df-orchestrate parses implementation-agent results as JSON at runtime; malformed results trigger sentinel handling without crashing the pipeline.
- **findings.md intermediary**: implementation-agent writes `{name}.findings.md` at runtime after architect review; code-agent probes for its existence at startup; file is deleted during Step 5 cleanup.
- **BLOCKED result path**: code-agent returns BLOCKED status at runtime when unresolvable ambiguity is encountered; implementation-agent reads this as a gate failure and reports to developer.
