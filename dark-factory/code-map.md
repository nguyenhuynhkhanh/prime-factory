## Circular Dependencies

None detected. The dependency graph is a DAG:
- Skills spawn agents (never the reverse)
- Agents read shared data files (project-profile, project-profile-slim, code-map, code-map-slim, manifest, promoted-tests, memory/) but do not import each other
- implementation-agent spawns other agents but is never spawned by them (only by df-orchestrate)
- codemap-agent is spawned by onboard-agent and pre-pipeline hooks but never spawns non-scanner agents
- spec-agent can be re-spawned by implementation-agent (for architect blockers OR flaky-E2E bugfix) but does not spawn implementation-agent — still acyclic
- architect-agent reads DI shards and slim files but never spawns codemap-agent (which produces them)
- promote-agent writes to memory/ (including DI shards) but is never spawned by onboard-agent — both write to memory/ independently (bootstrap vs. ongoing), no cycle
- build-agents.js reads src/*.src.md and writes .claude/agents/ — no agent reads its own source; no cycle

## Dynamic/Runtime Dependencies

- **Agent spawning via Agent tool**: All inter-agent communication happens through Claude Code's Agent tool at runtime. Not statically traceable from file imports.
- **Tier-conditional architect spawn**: implementation-agent reads `Architect Review Tier` from spec at runtime. Decision to spawn 1 combined vs. 3 parallel domain architects is made per-spec.
- **Pipeline mode dispatch**: implementation-agent receives `--mode` at runtime; selects claude-sonnet vs. claude-opus for generator agents per mode+tier combination. Judge agents invariantly use claude-sonnet.
- **Best-of-N activation**: quality mode + Tier 3 → two parallel code-agent worktrees at runtime. Neither condition alone activates it.
- **Slim file fallback**: architect-agent probes for slim file existence at runtime (Tier 1/2 only). Falls back to full file silently if slim is absent.
- **Manifest-driven dispatch**: df-orchestrate reads manifest.json at runtime to determine which specs to implement and in what order.
- **Index-first memory shard selection**: spec-agent and debug-agent select which domain shards to load at runtime based on index.md tags. Missing index → fallback to all shards.
- **DI shard availability**: design-intent shard files may be absent (pre-onboard or not yet extracted). Any agent requesting a missing DI shard logs and continues — non-blocking.
- **Memory shard availability**: individual INV/DEC shard files may be absent (not yet written). Any agent requesting a missing shard logs and continues — non-blocking.
- **Serena MCP availability**: code-agent and debug-agent probe Serena tool availability at runtime and fall back to Read/Grep. debug-agent is read-only (never uses Serena mutation tools).
- **Plugin installation targets**: bin/cli.js copies files to arbitrary target directories at runtime.
- **Flaky-E2E routing decision**: implementation-agent reads `flakyE2E` from test-agent results metadata at runtime.
- **Dev server detection**: test-agent's webServer source is determined at runtime by probing playwright.config.*, then project-profile, then npm run dev.
- **Git hook target**: onboard-agent's hook target file is selected at runtime based on detected hook infrastructure in the target project.
- **AFK PR creation**: implementation-agent reads spec `## Context` and `## Acceptance Criteria` at runtime to draft PR body; gh pr create is a runtime shell call (non-blocking if it fails).
