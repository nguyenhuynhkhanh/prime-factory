## Cross-Cutting Concerns

- **Information barriers**: Every agent declares explicit NEVER-read constraints. Holdout scenarios are hidden from code-agent. Public scenarios are hidden from test-agent. Tests/scenarios are hidden from architect-agent. Enforced by 50+ test assertions.
- **Code map freshness protocol**: df-intake, df-debug, and df-orchestrate all share the same pre-pipeline hook pattern: read code-map.md header -> compare git hash -> spawn codemap-agent if stale.
- **Plugin mirroring**: all agents, skills, rules, and templates must exist in both `.claude/` and `plugins/dark-factory/`. Contracts test enforces exact match.
- **Worktree isolation**: df-intake, df-debug, df-orchestrate, and implementation-agent (multi-track code-agents) all spawn agents with `isolation: "worktree"` for parallel, conflict-free execution. Wave-based dependency resolution in df-orchestrate.
- **Template-driven output**: spec-agent, debug-agent, and onboard-agent all read structured templates from `dark-factory/templates/` rather than embedding output format inline.
- **Pre-flight test gate**: implementation-agent runs project test suite before architect review. Bypassable via `--skip-tests` (logged in manifest as `testGateSkipped: true`).
- **Parallel domain review**: implementation-agent spawns 3 architect-agents simultaneously (Security & Data Integrity, Architecture & Performance, API Design & Backward Compatibility). Synthesizes via strictest-wins logic; deduplicates overlapping findings; escalates contradictions to developer. Caches per-domain reviews to avoid re-spawning.
- **UI Layer / E2E detection (Phase 2.5)**: onboard-agent uses an explicit allowlist for frontend frameworks (react, vue, angular, next, nuxt, svelte, sveltekit, remix, gatsby, astro, solid, qwik, ember, lit) and E2E frameworks (playwright, cypress). Records 4 rows in profile Tech Stack table consumed by test-agent for backend-only exclusion.
- **Flaky E2E routing**: test-agent emits `flakyE2E: true` boolean in result metadata. implementation-agent uses this AS THE SINGLE AUTHORITATIVE SIGNAL — routes flaky scenarios to spec-agent in bugfix mode (does NOT re-run code-agent), routes clean failures to code-agent re-run. Mixed cases run both in parallel. Flaky scenarios do not count toward 3-round retry max.
- **Dev server lifecycle (test-agent)**: 3-tier fail-soft cascade for E2E dev server: (1) Playwright config webServer property (delegates to Playwright), (2) project-profile `Dev Server` command, (3) `npm run dev` fallback. Always cleans up via process group kill regardless of outcome.
- **Git hook installation (Phase 7.5)**: onboard-agent detects existing infrastructure (Husky, Lefthook, simple-git-hooks) and installs into the appropriate location with a `# dark-factory-hook` comment marker. Falls back to raw `.git/hooks/pre-commit`. Idempotent — skips if marker already present.

## Dynamic/Runtime Dependencies

- **Manifest-driven dispatch**: df-orchestrate reads manifest.json at runtime to determine which specs to implement and in what order. Dependency graph between specs and group membership are declared in manifest entries. Wave resolution is computed dynamically.
- **Serena MCP availability**: code-agent and debug-agent probe Serena tool availability at runtime and fall back to Read/Grep. Behavior depends on whether Serena server is running.
