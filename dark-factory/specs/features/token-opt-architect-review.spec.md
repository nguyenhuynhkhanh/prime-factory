# Feature: token-opt-architect-review

## Context

The architect-agent review pipeline currently spawns 3 domain architects × 3+ rounds for every spec, regardless of whether the spec touches 2 files or 20. A one-line change to a single agent file gets the same review overhead as a migration-touching, security-sensitive, cross-pipeline rewrite. This produces two failure modes:

1. **Over-review**: Simple specs burn 9 sessions (3 agents × 3 rounds) when 1 session would suffice, consuming significant token budget and calendar time.
2. **Context overload**: Every architect session loads the full project profile and code map even when the spec affects two files and has no cross-cutting concerns, wasting context window on irrelevant material.

This spec introduces three coordinated improvements to the architect-agent review pipeline:

1. **Tiered review** — adaptive round count AND agent count based on spec complexity (Tier 1 = 1 agent × 1 round; Tier 2 = 3 agents × 2 rounds; Tier 3 = 3 agents × 3+ rounds, unchanged).
2. **Slim context loading** — architect-agents at Tier 1/2 read slim context files generated by `token-opt-slim-context` rather than full project profile and code map by default.
3. **Round summarization** — compact handoff notes written to disk after each round, eliminating full conversation re-reads between rounds.

Together these changes reduce token consumption for simple specs (Tier 1) from 9 sessions to 1, while preserving the full review depth for complex specs (Tier 3).

## Scope

### In Scope (this spec)
- `architect-agent.md` (and plugin mirror): tiered round count logic, slim context loading protocol, round summarization protocol, domain parameter behavior for Tier 1 combined agent
- `implementation-agent.md` (and plugin mirror): tier-aware architect spawn (Tier 1 = 1 combined agent, Tier 2/3 = 3 domain agents), tier passed as spawn parameter
- `spec-agent.md` (and plugin mirror): Complexity Classification field added to spec output, user confirmation step for tier, self-assessment fallback instruction
- `dark-factory/templates/spec-template.md` (and plugin mirror): new `Architect Review Tier` field added to template
- `tests/dark-factory-setup.test.js`: replace single "minimum 3 rounds" assertion with 3 targeted assertions; update architect-agent token cap from 4,500 to 5,000

### Out of Scope (explicitly deferred)
- `onboard-agent.md` — handled in `token-opt-slim-context` spec
- `codemap-agent.md` — handled in `token-opt-slim-context` spec
- `df-cleanup/SKILL.md` — handled in `token-opt-slim-context` spec
- `project-profile-slim-template.md` — handled in `token-opt-slim-context` spec
- Generating `project-profile-slim.md` or `code-map-slim.md` — handled in `token-opt-slim-context` spec
- Round summarization for domain-specific reviews that are NOT yet implemented (this spec adds the protocol; the first implementation will exercise it)
- Metrics or observability for tier distribution across all features in a pipeline run
- Any changes to the test-agent, promote-agent, code-agent, or debug-agent

### Scaling Path
The tier system is a floor, not a ceiling. The architect self-escalates when it discovers unexpected complexity. If tier distribution data (from future observability tooling) shows that Tier 1 is routinely self-escalating to Tier 2, the classification thresholds can be tightened. The summarization file format is versioned by the path pattern (round{N}) so future expansions (e.g., structured JSON summaries) can coexist with the current markdown format.

## Requirements

### Functional

- FR-1: Tier classification field — The spec-agent MUST add an `Architect Review Tier` field to every spec it writes. This field captures the tier (1, 2, or 3) and the primary reason for the classification. The field appears in the spec output immediately after the `Implementation Size Estimate` section.
- FR-2: Tier classification by spec-agent — When completing a spec, the spec-agent MUST assess the spec's tier based on the classification signals table (see Business Rules) and populate `Architect Review Tier`. The spec-agent MUST ask the user to confirm the tier before finalizing the spec. If the user defers ("let architect decide"), the field is written as "Unset — architect self-assesses."
- FR-3: Tier-aware architect spawn — The implementation-agent MUST read the `Architect Review Tier` field from the spec before spawning architect-agents. For Tier 1: spawn 1 combined architect-agent (no domain parameter). For Tier 2/3: spawn 3 domain architect-agents in parallel as today. The tier value MUST be passed as a spawn parameter to each architect-agent session.
- FR-4: Architect round budgets — The architect-agent MUST honor the following round budgets by tier: Tier 1 = 1 round minimum (no follow-up unless blocker found); Tier 2 = 2 rounds minimum; Tier 3 = 3+ rounds minimum (unchanged from current behavior). A tier is a floor, not a ceiling — the architect can always run additional rounds if a blocker surfaces.
- FR-5: Architect self-assessment — When the `Architect Review Tier` field is "Unset" or the architect disagrees with the recorded tier, the architect MUST self-assess using the same classification signal table and document its assessed tier at the top of its review output. Self-assessed tiers use the strictest-wins rule: if any domain agent assesses a higher tier than the spec-recorded tier, the highest assessed tier governs.
- FR-6: Slim context loading — Tier 1 architect MUST read `dark-factory/project-profile-slim.md` and `dark-factory/code-map-slim.md` by default instead of full files. Tier 2 architect MUST read slim files by default but MUST fall back to full files if it encounters a gap it cannot reason about (and log the reason). Tier 3 architect MUST read full files. If a slim file is missing, the architect MUST silently fall back to the corresponding full file — never fail.
- FR-7: Round summarization — After each refinement round (Tier 1: after round 1 only if escalated; Tier 2: after each of rounds 1–2; Tier 3: after each of rounds 1–3+), the architect MUST write a compact handoff note to `dark-factory/results/{name}/review-{domain}-round{N}-summary.md`. The note MUST use the required structure (see Business Rules) and MUST NOT exceed 400 words / 600 tokens.
- FR-8: Handoff note reading — At the start of each round N > 1, the architect MUST read the round (N-1) summary note before reviewing the updated spec. For round 1 (no prior summary): the architect proceeds without a summary note. Missing summary files are treated as round 1 (graceful fallback — never fail).
- FR-9: Architect escalation — When a Tier 1 architect discovers complexity that warrants more review (e.g., a blocker reveals an undetected migration path or a security surface), it MUST self-escalate by: (a) documenting the escalation reason in its review output, (b) running additional rounds as needed, and (c) updating its findings to note "Escalated from Tier 1 to Tier {N}: {reason}". The implementation-agent reads this escalation note and records it in the manifest.
- FR-10: Combined architect role (Tier 1) — When spawned without a domain parameter AND the spec is Tier 1, the architect-agent performs a unified review covering all three domains (Security & Data Integrity, Architecture & Performance, API Design & Backward Compatibility) in a single session. It produces a single `{name}.review.md` file (not domain-specific files).
- FR-11: Spec-template field — `dark-factory/templates/spec-template.md` and its plugin mirror MUST include the `Architect Review Tier` field definition with: allowed values (Tier 1 / Tier 2 / Tier 3 / Unset — architect self-assesses), format guidance, and a brief classification signal reference.
- FR-12: Test assertions — `tests/dark-factory-setup.test.js` MUST replace the single "minimum 3 rounds" assertion with three targeted assertions: (1) tiered language exists in architect-agent ("Tier 1", "Tier 2", "Tier 3" with round budget for each); (2) Tier 3 specifies minimum ≥ 3 rounds (safety floor preserved); (3) Tier 1 specifies exactly 1 round as its budget. The architect-agent token cap test MUST be updated from 4,500 to 5,000 tokens.

### Non-Functional

- NFR-1: Information barrier preservation — The tiering system MUST NOT create any pathway where the architect-agent sees scenario content (public or holdout), test strategy, or test coverage discussion. Tier classification is based entirely on spec content: file count, migration section, security domain signals, cross-cutting keywords. Zero test content flows through the tier parameter or round summary.
- NFR-2: Slim file fallback is silent — If `project-profile-slim.md` or `code-map-slim.md` is missing, the architect falls back to the full file without surfacing an error to the developer. The fallback is logged internally ("Slim file not found, reading full file") but does not appear in the review output.
- NFR-3: Round summary budget — Round summary files MUST NOT exceed 400 words or 600 tokens. Architects MUST prioritize brevity: resolved items are listed, not explained; open blockers are stated, not justified.
- NFR-4: Plugin mirror sync — All changes to agent files and spec-template.md in `.claude/` MUST be mirrored to `plugins/dark-factory/`. The contracts test suite enforces exact content parity.
- NFR-5: Backward compatibility — Existing specs that do not have an `Architect Review Tier` field (pre-feature specs) MUST be treated as Tier 3 (the current default). The implementation-agent MUST NOT fail on missing tier fields.

## Data Model

No database or persistent storage. This feature modifies markdown agent definition files and the spec template. The only new file type introduced is the round summary note:

**Round summary note path pattern:** `dark-factory/results/{name}/review-{domain}-round{N}-summary.md`
- `{name}`: spec name (matches the feature name in the manifest)
- `{domain}`: domain slug — `security`, `architecture`, `api`, or `combined` (for Tier 1 combined agent)
- `{N}`: round number (1, 2, 3, ...)

**Round summary note structure (required):**
```
Round {N} Summary ({domain}):
- Resolved this round: [list of issues addressed since last round]
- Open blockers (must address next round): [list, or "None"]
- Key decisions made: [list of binding decisions]
- Next round focus: [what to verify in the next round]
```

**Architect Review Tier field format (in spec output):**
```
## Architect Review Tier
- **Tier**: {1 | 2 | 3 | Unset — architect self-assesses}
- **Reason**: {primary signal that drove this classification, referencing the signal table}
- **Agents**: {1 combined | 3 domain agents}
- **Rounds**: {1 | 2 | 3+}
```

## Migration & Deployment

**Existing specs without `Architect Review Tier` field:** The implementation-agent MUST treat a missing tier field as Tier 3. This is backward compatible: existing behavior (3 agents × 3+ rounds) is preserved for all in-flight or archived specs.

**Agent files are read at spawn time:** No cached versions exist. Updating the `.md` files takes effect immediately on the next agent spawn. No stale data issue.

**Test assertion change:** The existing "minimum 3 rounds" assertion at line 254 of `tests/dark-factory-setup.test.js` is replaced in-place with three assertions. The existing architect-agent token cap test (cap: 4,500) is updated to 5,000. Both changes are in the same file, in the same test suite — no test registry updates needed.

**Deployment order:**
1. `token-opt-slim-context` spec must be deployed first (slim files must exist at deploy time for Tier 1/2 architects to load them).
2. This spec (`token-opt-architect-review`) can deploy immediately after.
3. No zero-downtime concern — Dark Factory is a prompt framework, not a running service.

## Business Rules

- BR-1: Tier classification signals

| Signal | Tier |
|--------|------|
| ≤ 2 files touched, no migration section, no security/auth domain, no cross-cutting keywords | Tier 1 |
| 3–4 files touched, OR some cross-cutting concerns but not Tier 3 triggers | Tier 2 |
| 5+ files touched, OR migration section populated, OR cross-cutting keywords present ("all agents", "pipeline", "system-wide"), OR touches shared templates or test contracts, OR security/auth domain | Tier 3 |

- BR-2: Tier is a floor — The architect can always self-escalate to a higher tier if it discovers complexity the spec-agent missed. It cannot self-downgrade (a Tier 3 spec stays at Tier 3 even if the architect finds it simpler than expected).
- BR-3: Strictest-wins for domain disagreement — When multiple domain architects self-assess conflicting tiers, the highest assessed tier governs for all subsequent rounds in that review.
- BR-4: Existing specs default to Tier 3 — Any spec missing the `Architect Review Tier` field is treated as Tier 3 (maximum review). This ensures no regression in review quality for legacy specs.
- BR-5: Round summary is mandatory after each completed round — A round that resolves blockers and produces a spec update MUST produce a round summary file before the next round begins. A round summary for round 1 of a Tier 1 spec that produces no blockers and no further rounds is still written (it records "Resolved this round: initial review complete, no blockers found").
- BR-6: Tier 1 combined agent covers all domains — The Tier 1 combined architect is not a "lightweight" review. It performs the same depth of analysis as 3 domain agents, just in one session. It MUST evaluate Security & Data Integrity, Architecture & Performance, and API Design & Backward Compatibility.
- BR-7: Slim context is a default, not a gate — Reading slim files does not preclude reading full files. A Tier 2 architect that hits a gap it cannot reason about from slim files MUST read the full file and log the reason. Slim files are a starting point, not a restriction.
- BR-8: Information barrier is absolute — Tier classification is based solely on spec content signals (file count, migration section, keywords, security domain). No scenario content, test strategy, or holdout content ever informs tier classification or round summaries.

## Error Handling

| Scenario | Response | Side Effects |
|----------|----------|--------------|
| Spec has no `Architect Review Tier` field | Treat as Tier 3, proceed with 3-domain parallel review | Log: "Architect Review Tier field missing — defaulting to Tier 3 (maximum review)" |
| Tier field is "Unset — architect self-assesses" | Architect self-assesses using classification signal table, records tier in review output | No spawn change — implementation-agent treats unset as Tier 3 for spawn decision; architect documents its self-assessment |
| Slim file (`project-profile-slim.md`) missing | Fall back to full `project-profile.md` silently | Log internally: "Slim file not found, reading full file" — not surfaced to developer |
| Slim file (`code-map-slim.md`) missing | Fall back to full `code-map.md` silently | Log internally: "Slim file not found, reading full file" — not surfaced to developer |
| Round N-1 summary file missing when starting round N | Proceed without summary (treat as first round) | Log internally: "No summary found for round N-1 — proceeding without handoff note" |
| Round summary exceeds 400 words | Truncate to fit, prioritize blockers and key decisions | Truncation is logged in the summary file itself as a trailing note |
| Tier 1 architect discovers escalation-worthy complexity | Self-escalate, run additional rounds, record escalation reason in review output | Implementation-agent records escalation in manifest; review takes longer than budgeted |
| Domain architects disagree on tier | Strictest-wins — highest assessed tier governs | All domains run additional rounds to match the highest tier's round budget |
| Tier 2 architect hits a reasoning gap in slim files | Read full file, log reason | "Slim file insufficient for {topic} — reading full {file}" — appears in review output |
| architect-agent token cap exceeded after tiering logic added | Fail test with clear message showing actual vs. expected tokens | Developer must trim architect-agent content to stay under 5,000 token cap |

## Edge Cases

- EC-1: Spec with missing `Architect Review Tier` field — treated as Tier 3, full 3-domain parallel review, no failure (backward compat)
- EC-2: Tier field present but value is unrecognized (e.g., "Tier 4" or "medium") — treat as Tier 3 (strictest default)
- EC-3: Tier 1 architect finds a blocker in round 1 — must self-escalate, run at least one additional round, write round 1 summary before round 2 begins
- EC-4: Round summary file for round N-1 is missing when round N starts — proceed gracefully without the note, do not fail the review
- EC-5: Both `project-profile-slim.md` AND `project-profile.md` are missing — architect proceeds with only code-map context (or slim code-map if available); logs the gap; does not block the review
- EC-6: Tier 2 domain architects each assess different tiers (one says Tier 2, one says Tier 3) — strictest-wins; all three run 3+ rounds to match the Tier 3 assessment
- EC-7: Round summary exceeds 400 words — truncated; blockers and key decisions preserved; truncation noted in the file itself
- EC-8: Implementation-agent receives a spec with tier "Unset — architect self-assesses" — spawn as if Tier 3 (3 domain agents); architect performs self-assessment and records tier in its output
- EC-9: Tier 1 architect self-escalates mid-review — escalation reason recorded in review output; implementation-agent records escalation in manifest; additional rounds run within the same session
- EC-10: `plugins/dark-factory/` mirror is out of sync with source — contracts test fails; deployment blocked until both sides are updated
- EC-11: architect-agent content exceeds 5,000 token cap after edits — setup test fails with actual vs. expected token count; developer must trim content before proceeding
