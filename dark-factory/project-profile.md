## Architecture
### Structure
Dark Factory is a prompt engineering framework with a dual-distribution model. It consists of:
1. **Agent definitions** (`.claude/agents/*.md`) — 9 markdown files with YAML frontmatter that define agent behavior, access rules, and processes
2. **Skill definitions** (`.claude/skills/*/SKILL.md`) — 8 slash command handlers that orchestrate agent spawning
3. **A CLI** (`bin/cli.js`) — installs/updates Dark Factory into target projects by copying from the plugin directory
4. **A plugin package** (`plugins/dark-factory/`) — self-contained copy of all agents, skills, rules, and templates for Claude Code plugin distribution
5. **Template files** (`dark-factory/templates/`) — structured output templates for specs, debug reports, and project profiles
6. **Two test suites** (`tests/`) — structural integrity tests (331 tests) and cross-feature contract tests
7. **Runtime directories** (`dark-factory/`) — where specs, scenarios, results, and promoted-tests tracking live during use

There is no "running application." The framework operates entirely through Claude Code's agent and skill system. The CLI is the only executable code.

### Patterns to Follow
- **Agent definitions**: Markdown with YAML frontmatter (`name`, `description`, `tools`). Body contains role description, process steps, templates, and constraints. Example: `.claude/agents/spec-agent.md`
- **Skill definitions**: Markdown with YAML frontmatter (`name`, `description`). Body contains trigger syntax, process steps, and rules. Example: `.claude/skills/df-intake/SKILL.md`
- **CLI**: `bin/cli.js` copies files from `plugins/dark-factory/` into target projects. No inline template generation — source of truth is the plugin directory.
- **Plugin mirroring**: Every agent, skill, rule, and template must exist in BOTH `.claude/` (for this repo) AND `plugins/dark-factory/` (for distribution). Tests enforce that plugin mirrors match source files.
- **Information barriers**: Every agent explicitly declares what it CANNOT access using "NEVER read/modify" constraint blocks. These are tested.
- **Frontmatter naming**: The `name` field in frontmatter MUST match the filename (without `.md`) for agents and the directory name for skills. Tests enforce this.
- **Template files**: Agents reference templates from `dark-factory/templates/` rather than embedding output structure inline. Templates for: spec, debug report, project profile.

### Shared Abstractions
- `parseFrontmatter()` in both test files — parses YAML frontmatter from agent/skill files (test-only utility)
- `copyFileWithLog()` / `copyDirRecursive()` in `bin/cli.js` — idempotent file installation with change detection
- Template files in `dark-factory/templates/` — structured output formats referenced by agents
- `promoted-tests.json` — registry tracking which holdout tests have been promoted to the permanent suite

## Structural Notes

- **Plugin mirror is the source of truth for distribution.** The `plugins/dark-factory/` directory contains the complete package that gets installed into target projects via `bin/cli.js`. Changes to agents/skills must be made in BOTH `.claude/` (for this repo) AND `plugins/dark-factory/` (for distribution). Tests enforce mirror consistency.

- **scripts/ directory is empty.** The old `init-dark-factory.js` script has been replaced by `bin/cli.js`. The `scripts/` directory remains but contains no files.

- **VERSION file in plugin.** `plugins/dark-factory/VERSION` contains the version string (currently `1.3.0`). The CLI reads this for version display and the plugin.json references it.

- **No CI/CD pipeline.** Tests must be run manually with `node --test tests/`.

- **The dark-factory/ subdirectory in this repo is mostly scaffolding.** It contains `manifest.json` (empty features object), `promoted-tests.json` (tracking promoted features), templates, and `.gitkeep` files. In a real target project, this directory accumulates specs, scenarios, and results during use.

## Common Gotchas

- **Dual-write requirement for agents/skills.** Any change to an agent or skill file must be made in both `.claude/agents/` (or `.claude/skills/`) AND `plugins/dark-factory/agents/` (or `plugins/dark-factory/skills/`). Forgetting one side causes test failures in the plugin mirror consistency suite.

- **Test file is very large.** `dark-factory-setup.test.js` is ~3000 lines. When adding promoted tests, append to the end of the file within the appropriate section markers (`DF-PROMOTED-START` / `DF-PROMOTED-END`).

- **No node_modules.** The project has zero npm dependencies. Do not add `require()` calls for external packages.

- **Frontmatter name must match filename.** Agent file `foo-agent.md` must have `name: foo-agent` in its YAML frontmatter. Tests enforce this strictly.

## For New Features
- **Adding a new agent**: Create the `.md` file in `.claude/agents/` AND copy it to `plugins/dark-factory/agents/`. Add frontmatter with `name`, `description`, `tools`. Add information barrier tests if the agent has access restrictions. Add plugin mirror test.
- **Adding a new skill**: Create `SKILL.md` in `.claude/skills/{skill-name}/` AND `plugins/dark-factory/skills/{skill-name}/`. Add frontmatter with `name`, `description`. Add plugin mirror test.
- **Adding a template**: Create the template in `dark-factory/templates/` AND `plugins/dark-factory/templates/`.
- **Modifying agent behavior**: Update BOTH the `.md` file in `.claude/agents/` and in `plugins/dark-factory/agents/`. Run tests to verify frontmatter, barrier constraints, and mirror consistency.
- **Test expectations**: Every agent must have valid frontmatter. Every information barrier must be tested. Plugin mirrors must match. Every new slash command must be documented in CLAUDE.md.

## For Bug Fixes
- **Running all tests**: `node --test tests/` (331 tests, ~160ms)
- **Running setup tests only**: `node --test tests/dark-factory-setup.test.js`
- **Running contract tests only**: `node --test tests/dark-factory-contracts.test.js`
- **Common failure patterns**: Frontmatter name mismatch (agent filename vs frontmatter `name` field), missing barrier phrases in agent content, plugin mirror out of sync with source files
- **Fragile areas**: Plugin mirror consistency (forgetting to update both sides), promoted test section markers (must match exactly), the dual-write pattern for any agent/skill change
